# AUDIT: atlas @ 667a13a (main)

**Verdict.** Atlas is a single-user, single-process Express + React application with SQLite persistence and an AI agent scheduler. The security floor is **mostly intact** â€” no hardcoded secrets, no `eval`, no command injection, no localStorage auth tokens, and CSP/cookie auth concerns don't apply since there's no auth layer yet. The most critical gaps are the **fully open CORS policy**, an **exposed directory-browser endpoint** (`/api/projects/ls`) that lets any caller enumerate any filesystem path on the host, and a **publicly writable notification insertion endpoint** with no input validation beyond two fields. There are no tests, no CI gates, and no structured logging â€” this is Tier 0/1 territory with Tier 1 promotion triggers already firing (real user data, public internet exposure via the dev proxy, agent-controlled LLM output fed back into system prompts).

## Findings â€” severity ranked

### CRITICAL

**C1 â€” Directory enumeration endpoint without path-traversal guard.** `server/routes/projects.ts:31-76` exposes `GET /api/projects/ls?path=<anything>`, which calls `fs.statSync(dirPath)` and `fs.readdirSync(dirPath)` on the raw user-supplied path with no canonicalization (`realpath`/`resolve` check) and no access scope. Any caller can enumerate arbitrary directories on the host: `/etc/passwd`, `/home/pefman/.ssh`, `/proc/self/environ`, etc. **Impact:** full host filesystem disclosure to any caller of the API. **Fix:** replace unguarded `fs.statSync`/`fs.readdirSync` with `realpath` canonicalization that rejects anything outside an allowlisted root (e.g. the project folder or the server cwd), or remove the endpoint and have the frontend use a native file picker.

**C2 â€” Public notification insertion without auth or input sanitization.** `server/routes/notifications.ts:34-51` exposes `POST /api/notifications` that inserts directly into the `notifications` table with only two required fields (`sender_role`, `message`). Any caller can craft notifications that impersonate any agent (`sender_role = 'ceo'`), inject HTML/markdown into user-facing notification UIs, or flood the table. **Impact:** UI spoofing, notification spam, potential XSS if the notification message is rendered unsanitized. **Fix:** gate behind authentication (even a simple bearer token), validate `sender_role` against the roles table, HTML-escape or sanitize `message`, and add rate limiting.

**C3 â€” Open CORS on all origins.** `server/index.ts:33` uses `app.use(cors())` with no `origin` restriction. Combined with C2 (public notification write), any web page on the internet can read all API data and write notifications. **Impact:** cross-origin data exfiltration + write abuse. **Fix:** restrict `cors()` to the frontend origin (`cors({ origin: process.env.FRONTEND_URL || 'http://localhost:5174' })`).

### HIGH

**H1 â€” Agent-controlled LLM output injected into system prompts.** `server/scheduler.ts:327` passes `ceoRole.system_prompt` and `task.description` as user/system messages to the LLM. The LLM's output is then parsed, used to insert subtasks into the DB (`scheduler.ts:378-381`), and later **fed back into the conversation as system prompt** for subtask execution (`scheduler.ts:462-465`). This is a prompt injection surface: a malicious or poorly-trained model can craft subtask output that, when stored as `outputs.content`, becomes the new subtask's system prompt on retry, or influence future CEO decompositions via the role's stored `system_prompt` content if the role is ever editable. **Impact:** unauthorized state mutation, data exfiltration via model output. **Fix:** validate and sanitize all LLM output before DB insertion â€” reject any JSON that doesn't strictly match the subtask schema (title/description/role/priority with enum validation), and never echo LLM output back into system prompts without review.

**H2 â€” Database path is relative to working directory with no protection.** `server/db.ts:7` uses `path.join(__dirname, '..', 'data', 'tasks.db')`. The `data/` directory is gitignored, which is good, but there is no check that the resolved path is within the project tree. A `__dirname` pointing outside the project (e.g. via symlink) could leak data to an arbitrary directory. **Impact:** data leakage if deployed with unusual file layout. **Fix:** `fs.mkdirSync` with an explicit `relativePath` check against `path.resolve(process.cwd(), 'data')`.

**H3 â€” Empty `setInterval` keep-alive pattern.** `server/index.ts:62` calls `setInterval(() => {}, 1000 * 60 * 60)` to "keep process alive." This is a known anti-pattern: it creates a timer handle the event loop must process every hour, leaks memory in some Node versions, and does nothing to prevent process exit from unhandled rejections or SIGTERM. The `process.on('unhandledRejection')` and `process.on('uncaughtException')` handlers exist but just log and exit â€” no graceful shutdown. **Impact:** resource waste, no graceful shutdown on SIGTERM (Cloud Run sends SIGTERM before eviction). **Fix:** remove the `setInterval`; rely on the event loop staying alive naturally (SSE connections, scheduler intervals). If graceful shutdown is needed, trap `SIGTERM`/`SIGINT`, stop the scheduler, close SSE connections, and exit.

**H4 â€” Notification SSE stream has no heartbeat.** `server/routes/notificationsStream.ts` opens an SSE connection but sends no heartbeat. If the client disconnects and reconnects, the server has no way to know the connection was lost until the next event fires. The kanban stream (`kanbanStream.ts:34-36`) correctly sends heartbeats every 25s; the notification stream does not. **Impact:** stale notification UI after network blips. **Fix:** add a periodic heartbeat to the notification SSE stream.

### MEDIUM

**M1 â€” No tests exist.** Zero test files, zero test configuration. The entire scheduler loop (parsing LLM output, managing retry timers, enqueuing subtasks) has no automated coverage. **Impact:** regressions in the execution loop go undetected. **Fix:** at minimum, add tests for the deterministic parsing functions (`parseToolCall`, `parseClarificationPayload`, `parseDecompositionSubtasks`, `normalizeSubtasks`) â€” these are pure logic with no I/O and are trivially testable.

**M2 â€” Unstructured console logging.** `server/lib/logger.ts` and AI providers use `console.log`/`console.error` with ad-hoc formatting. No JSON structured logs, no correlation IDs, no log levels beyond "log" vs "error." **Impact:** logs are unqueryable, unaggregatable, and impossible to correlate across request boundaries. **Fix:** switch to JSON structured logging with a per-request correlation ID (use `id` from `nanoid` or a simple counter), emitted to stdout.

**M3 â€” README is stale.** The README describes roles "Planner, Researcher, Writer, Reviewer" â€” none of which exist in the codebase. The project structure section lists `executor.ts` which doesn't exist (it's `scheduler.ts`). API endpoints listed are incomplete (missing messages, projects, repos, kanban stream, notifications stream). **Impact:** misleading documentation for anyone reading the codebase. **Fix:** rewrite README to match current code.

**M4 â€” `package.json` has no `engines` field and no scripts for lint/typecheck/audit.** There's no `lint`, `typecheck`, or `audit` script in `package.json`, so these checks can't be run as one command or as a CI gate. **Impact:** no automated quality gates. **Fix:** add `lint`, `typecheck`, and `audit` scripts.

**M5 â€” Settings endpoint stores API key as plaintext in SQLite.** `server/db.ts:173` creates `api_key TEXT` column; `server/routes/settings.ts:36-37` reads and writes it to/from the DB with no encryption. **Impact:** if the DB is compromised, the OpenAI API key is exposed in plaintext. **Fix:** at minimum, document this as a known limitation; ideally, use environment variables for the API key and never store it in the DB, or encrypt at rest.

**M6 â€” Activity log truncates output silently.** `server/routes/activity.ts:32` does `log.output?.substring(0, 200)` â€” silently truncating all execution output to 200 characters. This makes the "activity" endpoint almost useless for debugging (subtask outputs are typically hundreds or thousands of characters). **Impact:** degraded observability. **Fix:** return full output or paginate with a reasonable limit.

**M7 â€” Retry timer leak on task deletion.** `scheduler.ts:93-116` iterates `retryTimers` when removing a task â€” but if a subtask is deleted *between* the queue scan and the timer clear, the timer continues running and calls `this.enqueue()` on a non-existent subtask. The `executeItem` method does check for deletion (`scheduler.ts:438-441`) and logs a skip, but the timer is still paid every `2**n * 1000ms`. **Impact:** wasted retries and timer handles. **Fix:** cancel the timer in the `executeItem` skip path.

### LOW / DEFERRED

**L1 â€” No auth layer at all.** This is a single-user local app, so auth is out of scope for Tier 0/1. However, if Atlas is exposed to a network (it binds to `0.0.0.0`), there's zero protection. **Trigger:** any public exposure â†’ promote to Tier 2 and add auth.

**L2 â€” No structured logging format.** Console output is human-readable prose. Fine for local dev; not for production. **Trigger:** deploy outside localhost â†’ structured JSON logs.

**L3 â€” No backup story for SQLite.** `data/tasks.db` is a single file on disk with no backup mechanism. **Trigger:** any data loss event â†’ implement WAL archiving or periodic copy.

**L4 â€” No input sanitization on task/subtask titles and descriptions.** User-provided strings go into the DB and are rendered in the UI without escaping. With no XSS-unsafe rendering in the current UI (React auto-escapes), this is low risk but becomes a concern if markdown rendering is added.

## Strengths (verified)

- **No hardcoded secrets.** API keys are user-configurable and stored in the DB only when the user enters them. No keys in source.
- **Clean TypeScript compilation.** `tsc --noEmit` passes with zero errors â€” the type system is consistent.
- **No `eval`/`innerHTML`/`new Function`.** No DOM injection or code execution of user input.
- **`.env` is gitignored.** No risk of secret leakage via version control.
- **LLM output is parsed, not executed.** The scheduler extracts JSON from model output without calling it â€” prompt injection can't become code execution directly.
- **SQLite WAL mode enabled.** `db.pragma('journal_mode = WAL')` gives good concurrency for a single-process app.
- **SSE heartbeats on kanban stream.** `kanbanStream.ts:34-36` sends heartbeats every 25s â€” the notification stream should match this.
- **Uncaught exception handler exits cleanly.** `process.on('uncaughtException', ...)` logs and exits with code 1 â€” no silent corruption.
- **Subtask retry with exponential backoff.** The scheduler implements bounded retry (`MAX_ATTEMPTS = 3`) with capped backoff (`Math.min(2^n * 1000, 300000)`).
- **`npm audit` clean.** No known vulnerable dependencies.
